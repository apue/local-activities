# Snapshot and Refs

Use this file for ref-driven automation, annotated screenshots, and selector fallbacks.

**Related:** [commands.md](commands.md), [../SKILL.md](../SKILL.md)

## Why Refs Matter

Refs like `@e1` and `@e2` are compact handles generated by `snapshot` or `screenshot --annotate`. They are usually the fastest, lowest-token way for an AI agent to interact with a page.

Typical flow:

```bash
agent-browser open https://example.com
agent-browser wait --load networkidle
agent-browser snapshot -i
# inspect refs
agent-browser click @e3
agent-browser snapshot -i
```

## Snapshot Modes

```bash
agent-browser snapshot                # Full accessibility tree
agent-browser snapshot -i             # Interactive elements only
agent-browser snapshot -i --urls      # Include link targets
agent-browser snapshot -c             # Compact tree
agent-browser snapshot -d 4           # Limit depth
agent-browser snapshot -s "#main"    # Scope to a region
```

Use the smallest snapshot that still lets you act:
- `-i` for forms, menus, and buttons
- `-i --urls` when link destinations matter
- `-c` and `-d` to reduce token usage
- `-s` to focus on a specific panel, form, or modal

## Annotated Screenshots

Use this when text structure is not enough.

```bash
agent-browser screenshot --annotate ./page.png
# labels [1], [2], [3] map to refs @e1, @e2, @e3
agent-browser click @e2
```

Annotated screenshots help when:
- buttons are icon-only,
- layout matters,
- the page includes canvas or chart content,
- or you need spatial reasoning.

Notes:
- Refs are cached after an annotated screenshot, so you can act immediately.
- Annotated screenshots are supported on Chromium and Lightpanda, not the Safari/WebDriver path.

## Ref Lifecycle

Refs are valid only for the current page state. Re-snapshot after:

- navigation,
- form submission,
- tab or frame switches,
- opening a modal or dropdown,
- closing a modal,
- dynamic list updates,
- or any action that changes the DOM.

```bash
agent-browser snapshot -i
agent-browser click @e5
agent-browser snapshot -i   # required if the page changed
```

If you get a “ref not found” or an unexpected action target, assume the refs are stale and snapshot again.

## Best Practices

### Prefer refs first

```bash
agent-browser snapshot -i
agent-browser fill @e1 "user@example.com"
agent-browser click @e2
```

### Re-snapshot after page changes

```bash
agent-browser click @e4
agent-browser wait --text "Settings"
agent-browser snapshot -i
```

### Scope snapshots on large pages

```bash
agent-browser snapshot -s "#checkout" -i
agent-browser snapshot -s ".modal" -i -c
```

### Verify before acting on destructive controls

Use `get text`, `get attr`, or an annotated screenshot if the label is ambiguous.

## When Refs Are Not Enough

Use semantic locators when:
- the page changed before you could act,
- you want a stable label-based target,
- or the element is easier to describe semantically than visually.

```bash
agent-browser find role button click --name "Submit"
agent-browser find label "Email" fill "user@example.com"
agent-browser find testid "save-button" click
```

Use CSS selectors when the DOM structure is stable and already known.

## Troubleshooting

### Too many elements in the snapshot

```bash
agent-browser snapshot -i -c -d 4
agent-browser snapshot -s "#main" -i
```

### Element is missing

- Scroll it into view with `scroll` or `scrollintoview`.
- Wait for it with `wait <selector|@ref>` or `wait --text`.
- If it is inside an iframe, switch frames first.

```bash
agent-browser frame "#embedded-login"
agent-browser snapshot -i
```

### Page is visually clear but snapshot is not

Use an annotated screenshot instead:

```bash
agent-browser screenshot --annotate ./debug.png
```

### Need text extraction, not interaction

Prefer:

```bash
agent-browser get text body
agent-browser get text "#content"
```

This is usually smaller and faster than a full snapshot.
